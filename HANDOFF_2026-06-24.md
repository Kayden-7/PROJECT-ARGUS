### Key engineering decisions (with the "why")
- **Part 6 — kept deferred execution-creation (did NOT relocate into `approve()`).**
  A GPT-5.5 stress test showed relocating creation opens a cancel-vs-worker
  **double-send race**. Cross-checking the live Phase 5 tests confirmed the deferred
  model is safer. `approve()` now atomically stamps `approval_generation` + the live
  `approval_epoch` and refuses under hard stop; `promote_approved()` carries those onto
  the execution. Invariant held: **at most one Gmail send per queue item, ever.**
  (Spec SSOT updated in `47bdaf1`.)
- **Part 6 — executor preflight (`_preflight_hold`)** makes Emergency Stop actually
  halt in-flight executor work (a claim the pitch makes): hard stop / stale epoch →
  `HELD`, `owner_token` cleared so the row stays reopen-eligible.
- **Part 6 — R-REOPEN is state-branched** so it can never resurrect an ambiguous
  delivery: SENDING/MANUAL_REVIEW → 409; unclaimed pre-send → Fence A supersede;
  FAILED → reopen without supersede; COMPLETED → reconcile to EXECUTED;
  **CANCELLED is terminal** (closes the cancel→reopen double-send path).
- **Part 7 — reason cap split caller vs system.** Operator reasons are **rejected**
  (never truncated) over 500 chars / on NUL → `REJECTION_REASON_TOO_LONG`. System
  reasons are **degraded explicitly** (`…[+N chars]` marker) and can never fail the
  CHECK — the safety-critical quarantine write must always route.
- **Part 7 — demo reset** now refuses while a send is in flight (`EXECUTION_IN_FLIGHT`),
  reseeds exactly one protected contact fixture (`legal@confidential-client.com`) so
  the demo can SHOW the private-contact block, and writes a `DEMO_RESET_COMPLETED`
  genesis event to anchor the wiped audit chain.
